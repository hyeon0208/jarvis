# 아키텍처

## 전체 구조

Jarvis는 **Claude Code 위에 얹는 확장**입니다. 독립 프로그램이 아니라, Claude Code의 스킬/훅/MCP 시스템을 활용합니다.

```
┌──────────────────────────────────────────────────────────────┐
│                     Owner의 로컬 머신                          │
│                                                              │
│  ┌─────────────┐     ┌──────────────────────────────┐       │
│  │ jarvis chat │     │ Jarvis Daemon (백그라운드)      │       │
│  │ (터미널)     │     │ Telegram 봇 polling            │       │
│  └──────┬──────┘     └──────────┬───────────────────┘       │
│         │                       │                            │
│         ▼                       ▼                            │
│  ┌──────────────────────────────────────────────────┐       │
│  │              Claude Code (구독 기반)               │       │
│  │  ┌────────┐  ┌─────────┐  ┌───────────────┐     │       │
│  │  │ Skills │  │  Hooks  │  │  MCP Servers  │     │       │
│  │  └────────┘  └─────────┘  └───────────────┘     │       │
│  └──────────────────────────────────────────────────┘       │
└──────────────────────────────────────────────────────────────┘
         ▲                       ▲
         │                       │
      Owner가                 팀원들이
      직접 대화              Telegram으로 요청
```

## 두 가지 사용 모드

### 1. 직접 대화 모드 (`jarvis chat`)

```
Owner 터미널에서:
  jarvis chat → Claude Code 세션 시작 (Jarvis 성격 주입)
  
  사용자: "JPA N+1 문제 해결해줘"
  Jarvis: [IntentGate 분석 → 스킬 매칭 → 메모리 프리로딩 → 작업]
```

- 데몬 필요 없음
- 혼자 쓸 때 이것만으로 충분

### 2. 게이트웨이 모드 (`jarvis start`)

```
팀원 모바일 → Telegram @jv_gw_bot → Jarvis Daemon → Claude
                                        │
                    ┌───────────────────┼──────────────────┐
                    │                   │                  │
              인증 확인           프로필 권한 매핑       응답 전송
              (페어링 코드)      (--allowedTools)    (sendMessage)
```

- 팀원이 Telegram으로 원격 접근
- 백그라운드에서 24시간 동작

## 왜 Telegram 봇인가?

**Owner의 로컬 머신에 포트를 열지 않고도 외부 요청을 받기 위해서**입니다.

```
❌ 포트 오픈 방식:
   팀원 → ngrok/포트포워딩 → Owner:3000 → Claude
   문제: IP 노출, 보안 위험, 라우터 설정 필요

✅ Telegram 봇 방식:
   팀원 → Telegram 서버 ← Owner의 데몬이 1초마다 polling
   장점: 포트 오픈 없음, 보안 안전, 설정 간단
```

Telegram이 **메시지 큐** 역할을 합니다. Owner의 머신은 나가는 요청(polling)만 하고, 들어오는 연결은 없습니다.

## MCP 서버 역할

Jarvis는 2개의 MCP 서버를 사용합니다:

```
jarvis-memory (메모리 엔진)
├── 선언적 메모리 — 사실/선호도 저장
├── 절차적 메모리 — 스킬 인덱스
├── 세션 검색   — 과거 대화 FTS5 검색
├── Dreaming    — 메모리 정리/통합
└── 유저 프로파일 — 사용자 정보 관리

jarvis-gateway (채널 게이트웨이)
├── 메시지 라우팅 — 인증/권한 체크
├── DM 페어링    — 유저 인증
├── 프로필 관리  — 권한 설정
├── 크론잡      — 반복 작업
├── 샌드박스    — Docker 격리 설정
└── 워크트리    — git worktree 관리
```

## 보안 계층

```
1층: DM 페어링         — 승인된 유저만 접근
2층: 프로필 권한        — 유저별 도구 제한 (--allowedTools)
3층: 비용 제한          — 요청당 $0.5 제한 (admin 제외)
4층: Docker 샌드박스    — 파일시스템/네트워크 격리 (선택)
5층: 시간 제한          — 프로필별 타임아웃 (1~10분)
```
