# 아키텍처

## 전체 구조

Jarvis는 **Claude Code 위에 얹는 확장**입니다. 독립 프로그램이 아니라, Claude Code의 스킬/훅/MCP 시스템을 활용합니다.

```
┌──────────────────────────────────────────────────────────────┐
│                     Owner의 로컬 머신                          │
│                                                              │
│  ┌─────────────┐     ┌──────────────────────────────┐       │
│  │ jarvis chat │     │ Jarvis Daemon (백그라운드)      │       │
│  │ (터미널)     │     │ Telegram 봇 polling            │       │
│  └──────┬──────┘     └──────────┬───────────────────┘       │
│         │                       │                            │
│         ▼                       ▼                            │
│  ┌──────────────────────────────────────────────────┐       │
│  │              Claude Code (구독 기반)               │       │
│  │  ┌────────┐  ┌─────────┐  ┌───────────────┐     │       │
│  │  │ Skills │  │  Hooks  │  │  MCP Servers  │     │       │
│  │  └────────┘  └─────────┘  └───────────────┘     │       │
│  └──────────────────────────────────────────────────┘       │
└──────────────────────────────────────────────────────────────┘
         ▲                       ▲
         │                       │
      Owner가                 팀원들이
      직접 대화              Telegram으로 요청
```

## 두 가지 독립 사용 모드

`jarvis chat`과 `jarvis start`는 **역할이 다른 독립 명령**입니다. 둘 중 하나만 쓸 수도 있고, 병행할 수도 있습니다.

### 1. 직접 대화 모드 (`jarvis chat`)

```
Owner 터미널에서:
  jarvis chat
    → ① jarvis doctor --quick --silent-ok 자가 진단 (문제 있을 때만 출력)
    → ② Claude Code 세션 시작 (Jarvis 성격 + 자기 정보 + IntentGate 절차 주입)

  사용자: "JPA N+1 문제 해결해줘"
  Jarvis: [IntentGate 분석 → 스킬 매칭 → 메모리 프리로딩 → 작업]
```

- 포그라운드 대화 세션 (Owner가 터미널 앞에 있을 때)
- 데몬 없이 단독 동작
- 봇 토큰 불필요
- 자기 자신에 대한 질문이 들어오면 `~/jarvis/CLAUDE.md`와 `docs/00-overview.md`를 우선 참조하도록 지시
- 진입 시 진단을 건너뛰려면 `JARVIS_SKIP_DOCTOR=1 jarvis chat`

### 2. 게이트웨이 모드 (`jarvis start`)

```
팀원 모바일 → Telegram 봇 → Jarvis Daemon → claude -p 스폰
                                 │
                    ┌────────────┼──────────────┐
                    │            │              │
              인증 확인       프로필 매핑     응답 전송
              (페어링 코드)   (--allowedTools) (sendMessage)
```

- 백그라운드 데몬이 24시간 대기
- 외부 채널(Telegram/Discord/Slack) 메시지를 자동 처리
- Owner가 자리에 없어도 동작

### 3. 병행 사용

```
Owner 터미널:            # 포그라운드
  $ jarvis chat
  > 본인 작업 대화 중

Jarvis Daemon:           # 백그라운드
  팀원 A 메시지 수신 → claude -p 자식 프로세스로 처리
  팀원 B 메시지 수신 → 동시에 처리
```

Owner의 `jarvis chat`은 포그라운드 Claude Code 세션이고, `jarvis start`의 데몬은 별도 백그라운드 프로세스로 **서로 간섭하지 않습니다**. 자원이 충분하면 보통 둘 다 켜두고 씁니다.

## 왜 Telegram 봇인가?

**Owner의 로컬 머신에 포트를 열지 않고도 외부 요청을 받기 위해서**입니다.

```
❌ 포트 오픈 방식:
   팀원 → ngrok/포트포워딩 → Owner:3000 → Claude
   문제: IP 노출, 보안 위험, 라우터 설정 필요

✅ Telegram 봇 방식:
   팀원 → Telegram 서버 ← Owner의 데몬이 1초마다 polling
   장점: 포트 오픈 없음, 보안 안전, 설정 간단
```

Telegram이 **메시지 큐** 역할을 합니다. Owner의 머신은 나가는 요청(polling)만 하고, 들어오는 연결은 없습니다.

## MCP 서버 역할

Jarvis는 2개의 MCP 서버를 사용합니다:

```
jarvis-memory (메모리 엔진)
├── 선언적 메모리 — 사실/선호도 저장
├── 절차적 메모리 — 스킬 인덱스
├── 세션 검색   — 과거 대화 FTS5 검색
├── Dreaming    — 메모리 정리/통합
└── 유저 프로파일 — 사용자 정보 관리

jarvis-gateway (채널 게이트웨이)
├── 메시지 라우팅 — 인증/권한 체크
├── DM 페어링    — 유저 인증
├── 프로필 관리  — 권한 설정
├── 크론잡      — 반복 작업
├── 샌드박스    — Docker 격리 설정
└── 워크트리    — git worktree 관리
```

## 보안 계층

```
1층: DM 페어링         — 승인된 유저만 접근
2층: 프로필 권한        — 유저별 도구 제한 (--allowedTools)
3층: 비용 제한          — 요청당 $0.5 제한 (owner 제외)
4층: Docker 샌드박스    — 파일시스템/네트워크 격리 (선택)
5층: 시간 제한          — 프로필별 타임아웃 (1~10분)
6층: 메모리 격리        — JARVIS_USER_ID 환경변수로 채널/유저별 자동 분리
                         (자세히는 06-memory.md의 "사용자별 메모리 격리")
```
